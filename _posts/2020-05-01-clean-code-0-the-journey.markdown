### Coding for computers is easy. The challenge is to code for people.

In this series of posts, I present 5 high level rules and aspects that I
 collected over the years that help me write better code daily and might
 also help you.

Also, an important book that influenced me a lot in this long learning
 is [_Clean Code_ (Robert C. Martin, 2009)](cleancode), which I highly
 recommend you to read.

# ... but before, a little bit of story
When I was ~13yo and started coding, I used to admire complex code and
 hidden tricks. I had an evaluation method: **the more experienced a
 programmer was, the more complex his code should be, and the less I
 could understand from it.** It seems silly, and in fact it is! But for
 me, complexity was like a challenge.

Years (and some maturity) later, at the beginning of my career as a
 programmer, I started doing code reviews as well as having my own code
 for review. And if that wasn't yet very clear that complexity should
 be avoided and readability is indeed a big concern, at that moment,
 it turned to be. People had to reach me to clarify questions about
 logic, about design, about every aspect... and **writing code that
 people do not understand sucks!**

I then started jumping lines, for example at the beginning and at the
 end of every method, as well as within, for people reading the code
 better distinguish the ideas. Also, I started writing comments on my
 code, so it would be more understandable. I ended up writing comments
 for every line, which was a very big indication that my code didn't get
 any clearer, but only longer and more commented. Particularly relative
 to comments, I learnt within the years that **comments are like
 explanations for jokes**: *if you need then, it's big indication your
 code is not clear enough*.

During my experiences, I found many ways of improving my way of writing
 code. Some didn't work very well, but others were eventually more
 efficient. Either way, among the many factors that helped me develop a
 culture of writing more readable code, the **exposure to more
 experienced developers** is one of the most important facts **as well
 as the openness and willingness of being always criticised**.
